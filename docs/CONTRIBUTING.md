# Contributing to the Playbook

Thank you for helping improve how we work.

## Getting Started

New to the playbook? Start with [Developer Onboarding Guide](00-onboarding/developer_onboarding_guide.md).

Stuck on a technical issue? Check [Troubleshooting Guide](04-development/troubleshooting.md).

## Ways to Contribute

- **Fix a typo or broken link:** Open a PR directly.
- **Suggest a new section or process:** Open an **ADR (Architectural Decision Record)** if it's a significant change to our stack or workflow. Use the templates in `docs/03-design/adr/`.
- **Add practical examples:** Contribute starter kits, agent patterns, cloud vendor guidance, or compliance checklists.
- **Expand stack coverage:** Add docs for alternative backends, frontends, infrastructure platforms, or AI orchestration frameworks.
- **Report outdated information:** Open an Issue.

## 🌟 Star Maximization: How to Contribute Shareable Content

We want this playbook to be useful for the community and easy to star.

- Contribute diagrams, quickstart guides, or runnable templates.
- Add real-world examples that new users can clone and run immediately.
- Share case studies or usage notes in Discussions.
- Contribute Azure, on-prem, hybrid, or industry-specific guidance.
- Keep contributions concise, visual, and easy to link from `README.md`.

## PR Guidelines

1. Keep changes focused. One logical change per PR.
2. Update the `Last Updated` date in any file you modify.
3. If adding a new document, link it from the relevant index or the root `README.md`.
4. Request review from at least one member of the Kernel Team.

## Tone & Style

- Write for **clarity**, not completeness. Shorter is better.
- Use **active voice**. "The team deploys on Tuesday" not "Deployments are performed on Tuesday."
- Be **empathetic**. This is guidance, not a rulebook.

## Review Process

The Kernel Team reviews PRs weekly. If urgent, ping #eng-process.
